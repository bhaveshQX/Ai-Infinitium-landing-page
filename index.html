The problem

QA is the bottleneck.
Let's change that.

01
Tests take longer than the features they cover.
Engineering ships a feature in a sprint. The test suite catches up — if it does — a sprint later. Bugs reach production in the gap.
02
Selectors break. Nobody knows why.
A minor UI change breaks twenty tests. Locators were invented, not read from the live app. The fix costs a day you can't spare.
03
Requirements and tests live in different worlds.
Acceptance criteria sit in Jira. Tests sit in a repo. Keeping them aligned is manual work that gets skipped under pressure.

How it works

Eight stages.
Zero manual steps.

01
Fetch Requirements
Reads the Jira ticket — summary, acceptance criteria, priority — directly from source.
02
Understand Codebase
Uses QX Memory MCP to find Page Objects, fixtures, and call chains without reading raw files.
03
Crawl Live App
Agent opens the running app at runtime and collects real selectors. Never invents locators.
04
Build Test Cases
Maps each acceptance criterion to a concrete test case. Full list shown before any code is written.
05
Write Tests
Generates tests in your framework, following your existing file structure and code style.
06
Execute
Runs the real test runner — pytest, Playwright, JUnit — and captures the full output.
07
Report Results
Clear pass/fail summary with step-level detail on every failure.
08
Auto-Debug
Classifies each failure — app bug or script issue — traces root cause, fixes what it can.

Getting started

Up and running
in minutes.

STEP 01 Install & register MCPs
# install the package
pip install qx-memory
# register all skills + MCPs into Claude Code
qx setup-claude
# connect Jira (one-time, opens browser)
qx jira-auth
STEP 02 Index your project
# initialise + scan the codebase
qx init && qx scan
# build semantic embeddings for search
qx embed
# optional — visualize codebase as interactive diagram
qx mindmap
# optional — pre-crawl live app to cache UI locators
qx crawl <url>
During /generate-tests the agent crawls the app live via MCP. qx crawl is an optional pre-cache step only.
STEP 03 Generate test cases (end-to-end)
/generate-tests PROJ-123 --url http://localhost:3000
/generate-tests "login with MFA" --url http://staging.app
/generate-tests-selenium PROJ-456 --url http://localhost:8080
Fully automated: reads Jira → understands codebase → crawls app → writes tests → executes → debugs.
OPTIONAL Run specific tests only
/run-tests ./tests/e2e/login/ --url http://localhost:3000
/run-tests-selenium ./tests/test_login.py --url http://localhost:8080
Use when you want to re-run existing tests without regenerating. Includes auto-debug on failure.
